本文档审查文件：`src/main/resources/static/task1_classifier.py`

## 结论

该脚本可以完成最基础的分类调用，但目前存在多处明显风险，其中既包括安全问题，也包括稳定性和结果正确性问题。下面按严重程度排序。

## 1. 严重：API Key 硬编码在源码中

- 位置：`task1_classifier.py:11`
- 问题说明：
  脚本把 `openai.api_key` 直接写死在源码里。只要代码被提交到仓库、发给同学、上传到作业平台，密钥就会一起泄露。
- 影响：
  1. 任何拿到源码的人都可以直接使用该密钥调用接口。
  2. 会造成额度盗刷、费用损失，严重时还需要紧急废弃并更换密钥。
  3. 一旦进入版本历史，即使后面删掉，泄露风险仍然存在。

## 2. 高：没有任何异常处理，批处理容易中途整体失败

- 位置：`task1_classifier.py:24-32`，`task1_classifier.py:42-49`
- 问题说明：
  调用模型接口时没有捕获网络异常、认证异常、限流异常或返回结构异常。`batch_classify` 里也没有单条容错逻辑，只要其中一次请求失败，整个批处理就会直接中断。
- 影响：
  1. 批量任务运行到一半时可能因为一条数据失败而全部停止。
  2. 已处理结果只保存在内存中，异常发生时可能一个结果文件都拿不到。
  3. 在真实客服场景下，这会导致任务不稳定，无法可靠地交付分类结果。

## 3. 高：分类结果没有校验，模型输出一旦偏离标签集会直接写入结果

- 位置：`task1_classifier.py:16-32`
- 问题说明：
  脚本虽然在 prompt 中写了分类类别，但没有对返回值做任何校验。只要模型输出了“退款退货类”“账号相关问题”“这条属于物流查询”等非精确标签，程序也会原样写入 `predicted_category`。
- 影响：
  1. 输出结果可能不属于预定义的 6 个标签，导致下游路由或统计失败。
  2. 结果格式不稳定，后续如果做自动评测、聚合分析、数据库入库，都容易出错。
  3. 即使模型大部分时候答对，也缺少程序层面的兜底，结果不可控。

## 4. 中：分类规则没有复用权威定义文件，代码与规则文档可能逐渐不一致

- 位置：`task1_classifier.py:16-22`
- 问题说明：
  项目里已经有 `task1_categories.md`，其中明确写了分类定义和边界规则，比如“退款进度查询”应归入“退款退货”。但脚本没有读取或复用这份规则，只在 prompt 里写了一行简化版类别名称。
- 影响：
  1. 模型拿不到完整判定标准，容易把边界样本分错。
  2. 文档更新后，代码中的 prompt 不会自动同步，维护成本高。
  3. 代码行为和项目说明可能出现冲突，评测结果不稳定。

## 5. 中：逐条串行调用模型，性能和成本都偏差

- 位置：`task1_classifier.py:42-49`
- 问题说明：
  当前实现对每一条问题都单独发起一次模型请求，没有做批量化、并发控制、缓存或结果复用。
- 影响：
  1. 数据量一大时运行时间会明显变长。
  2. 请求次数过多，更容易触发限流或失败。
  3. 相比更合理的批量处理方式，调用成本会更高。

## 总体评价

如果这是课堂练习，这份代码说明作者已经掌握了最基本的接口调用流程；但如果把它当成可提交、可复用的工程代码，目前还不够安全，也不够稳健。最优先应该先解决密钥泄露和异常处理问题，其次补上标签校验与规则对齐。
